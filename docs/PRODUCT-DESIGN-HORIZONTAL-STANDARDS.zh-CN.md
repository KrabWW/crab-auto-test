# Crab 产品设计横向规范与检查表 v1

状态：已访谈确认，待后续页面设计继续细化。

本文档记录 Crab / RJ Auto Test 在进入具体页面设计前已经确认的横向规范。它不是某一个页面的设计稿，而是所有页面、组件、文案和交付物的共同判断标准。

## 1. 范围

本规范优先覆盖以下内容：

1. 视觉设计清单。
2. 组件设计清单。
3. 内容设计清单。
4. 设计系统与交付清单。
5. 页面检查表：UX、UI、视觉、交互、交付、证据可信。
6. 设计产物顺序。

UI 信息架构、一级导航、三栏工作台、参考产品映射已经在 `DESIGN.md`、`docs/REFERENCE-PRODUCT-PAGE-MAPPING.md` 和 `docs/UI-COMPONENT-BASELINE.md` 中沉淀，本文不重复展开。

## 2. 视觉设计清单

Crab 的视觉气质确定为高密度专业工作台，不走 AI 炫彩 SaaS 风格。

参考来源：

- shadcn/ui Blocks：https://ui.shadcn.com/blocks
- Atlassian Design：https://atlassian.design/
- Radix Themes：https://www.radix-ui.com/themes
- Mobbin Web：https://mobbin.com/explore/web

Codex 风格可以参考，但只用于 AI / Agent 执行层，例如右侧 AI 助手、执行过程、MCP tool call、自动化生成日志。Codex 的任务线程和过程透明感可以借鉴，但不能替代 Crab 的主产品结构。Crab 的主结构仍然是状态流转、证据矩阵、人工确认和 Proof Case 工作台。

### 2.1 视觉气质

确认规则：

- 采用专业、克制、高密度、可信、冷静、工程化、可审计的工作台气质。
- 不做大面积 AI 渐变、装饰光效、营销式大标题、低密度卡片页。
- 卡片只用于概览指标、摘要、推荐项和重复项，不承载主要工作流。
- 主工作流使用左侧对象/阶段导航、中间主工作区、右侧上下文/证据面板。

判断理由：

Crab 的核心不是展示 AI 很智能，而是证明一个需求的业务状态变化成立。视觉要让 QA、开发和发布负责人相信这个系统专业、可审计、能长期用于生产工作。

### 2.2 颜色原则

确认规则：

- 默认界面使用中性色。
- 颜色只服务状态、风险和责任归属，不做装饰性主色。
- 避免一个品牌蓝或品牌紫在所有页面里刷存在感。

语义色方向：

| 语义 | 颜色方向 | 用途 |
| --- | --- | --- |
| AI 建议 | 蓝 / 紫，低饱和 | AI 草案、AI 解释、AI 修复建议 |
| 人工确认 | 绿色 | 已确认、已批准、人工兜底通过 |
| 待确认 | 橙色 | 需要 QA / Owner 判断 |
| 证据不足 | 黄色 / Amber | 缺证据、证据弱、待补充 |
| 风险 / 失败 | 红色 | 状态未成立、执行失败、高风险 |
| 系统采集 | 灰蓝 | trace、日志、截图、接口响应、DB 快照 |
| 中性信息 | 灰色 | 元信息、未激活状态、辅助说明 |

判断理由：

Crab 页面里会同时出现 AI 是否可信、证据是否足够、状态是否成立、风险是否可接受、谁确认过等判断。如果颜色用于装饰，用户会分不清颜色到底代表什么。

### 2.3 字体字号

确认规则：

- 使用紧凑型专业工具字号体系。
- 正文 14。
- 表格 13。
- 代码 / 日志 / JSON / SQL 为 12 或 13。
- 页面标题 18 / 20。
- 避免官网式大标题和大留白。

字体方向：

| 场景 | 字体方向 |
| --- | --- |
| 中文界面 | PingFang SC / Microsoft YaHei / HarmonyOS Sans / Source Han Sans |
| 英文与数字 | Inter / SF Pro / Roboto |
| 代码、日志、JSON、SQL | JetBrains Mono / Fira Code / Consolas / Menlo |

判断理由：

Crab 会长期展示状态流转表、证据矩阵、API 请求响应、日志、trace、执行步骤和 AI 解释。字号太大，会降低 QA 审查效率。

### 2.4 间距、圆角、卡片

确认规则：

| 项 | 建议 |
| --- | --- |
| 间距体系 | 4 / 8 / 12 / 16 / 24 / 32 / 40 |
| 页面边距 | 24 |
| 模块间距 | 24 |
| 卡片间距 | 16 |
| 表格单元格 | 8 / 12 |
| 圆角 | 6 或 8 |
| 阴影 | 尽量少，用边框和背景层次替代 |

判断理由：

主要工作流如果大量使用卡片，页面密度会下降，层级也会混乱。Crab 应该优先采用紧凑间距、低装饰层级和轻圆角。

### 2.5 动效

确认规则：

- 只使用功能性动效。
- 不使用装饰性、循环性、营销式动效。

可以使用动效的地方：

- AI 助手抽屉打开 / 收起。
- 执行步骤从 queued 到 running 到 passed / failed。
- 日志、tool call、AI 输出流式追加。
- 证据详情展开。
- loading 到 result 的状态切换。
- 失败定位时高亮当前步骤。

不建议使用动效的地方：

- 背景光效。
- AI 粒子动画。
- 大面积渐变流动。
- 为科技感而存在的循环动画。
- 每个卡片 hover 都夸张弹跳。

判断理由：

动效应该帮助用户理解状态变化和系统正在做什么，而不是干扰 QA 看证据、日志和失败原因。

## 3. 组件设计清单

组件体系分三层：基础组件、业务组件、证据/状态组件。

| 层级 | 内容 | 来源 |
| --- | --- | --- |
| 基础组件 | Button、Input、Select、Table、Dialog、Sheet、Tabs、Badge、Tooltip | shadcn-vue / Radix / Tailwind |
| 业务组件 | Proof Case 状态卡、AI 草案卡、人工确认条、状态流转表、API 请求面板、执行时间线、报告结论卡 | Crab 自己沉淀 |
| 证据/状态组件 | 证据矩阵、证据详情抽屉、风险标签、状态标签、失败原因块、tool call 记录 | Crab 核心资产 |

判断理由：

Crab 真正的复杂度不在按钮和输入框，而在 Proof Case、状态流转、证据、AI 和人工确认。只做基础组件会让产品像拼出来的后台；沉淀业务组件，产品才会形成自己的专业语言。

### 3.1 弹框与抽屉规则

确认规则：

| 场景 | 组件 |
| --- | --- |
| 新建 / 编辑少量字段 | Dialog |
| 删除、禁用、吊销、清空、Danger Zone | Alert Dialog |
| AI 助手、证据详情、属性面板、检索测试、tool call 详情 | Right Sheet |
| 手机 / 窄屏下的右侧面板 | Drawer |
| API 请求编辑、Proof Case 详情、Trace 检查、证据矩阵 | 完整页面 / 三栏工作台 |

判断理由：

弹框适合短、明确、可完成的动作；抽屉适合在不离开当前页面的情况下查看上下文；复杂编辑器放进弹框会让用户迷失，也不利于审查证据。

### 3.2 表格优先规则

确认规则：

- 高频工作面优先使用 Data Table，不优先使用卡片列表。
- 需要筛选、排序、批量确认、风险比较、证据追踪的页面，优先 Data Table。
- 卡片只用于概览摘要、资产包、报告结论和少量推荐项。

适合表格的模块：

- 状态流转确认。
- 用例确认。
- 证据矩阵。
- API Case / Scenario 列表。
- 执行记录。
- MCP tools / invocation logs。
- Knowledge documents。
- Skills 列表。
- Settings 里的成员、密钥、审计日志。

判断理由：

Crab 的核心动作是比较、筛选、确认、追溯。卡片适合摘要，不适合大量字段对比。

### 3.3 状态标签体系

确认规则：

- Proof Case、Evidence、AI、Execution 四类状态分开建模。
- 所有状态必须走统一 Status Badge 体系。
- 禁止混用“通过 / 成功 / 完成”这类模糊状态。

状态组：

| 状态组 | 示例 |
| --- | --- |
| Proof Case 状态 | Draft、AI Generated、Needs Confirmation、Confirmed、Ready to Execute、Running、Evidence Insufficient、Proof Passed、Proof Failed、Report Generated、Asset Packaged |
| 证据状态 | Not Collected、Collecting、Collected、Valid、Invalid、Missing、Conflicting、Human Confirmed |
| AI 状态 | Generating、Suggested、Needs Review、Accepted、Edited Accepted、Rejected、Regenerating |
| 执行状态 | Queued、Running、Passed、Failed、Blocked、Timeout、Aborted |

状态标签必须统一：

- 文案。
- 颜色。
- 图标。
- Tooltip 解释。
- 是否可点击查看详情。
- 是否需要人工确认。

判断理由：

Passed 不等于 Proof Passed。AI Suggested 不等于 Human Confirmed。Evidence Collected 不等于 Evidence Valid。Case Passed 不等于 Business State Proved。这些差异必须通过状态体系表达清楚。

## 4. 内容设计清单

文案风格确定为事实型、结论型、可审计。文案要写清楚 AI 做了什么、系统采集了什么、人确认了什么、证据缺了什么、当前能不能下结论、下一步该做什么。

### 4.1 文案风格

不要写：

```text
AI 已为你智能完成全面检测！
```

应该写：

```text
AI 已生成 4 条状态流转建议，待人工确认。
```

不要写：

```text
测试成功！
```

应该写：

```text
状态流转成立：Disabled -> Enabled。已采集 UI、API、数据 3 类证据。
```

判断理由：

信任不是靠“智能”“全面”“高效”这些词，而是靠可验证事实。文案越具体，越可信。

### 4.2 错误提示结构

确认规则：

所有关键错误提示必须按四段式写：

1. 发生了什么。
2. 为什么失败。
3. 影响什么。
4. 下一步怎么做。

示例：

```text
发生了什么：未能证明状态流转成立。
为什么失败：页面显示已启用，但数据库 enabled 字段未变化。
影响什么：当前 Proof Case 不能标记为通过，报告会显示证据冲突。
下一步：检查启用接口是否只更新前端缓存，或补充数据层验证。
```

错误分类：

| 类型 | 例子 |
| --- | --- |
| 环境失败 | 无法访问测试环境 |
| 账号失败 | 测试账号无权限 |
| 脚本失败 | selector 失效 / 请求构造错误 |
| 状态失败 | 业务状态未变化 |
| 证据不足 | 只采集到 UI，没有 API / 数据证据 |
| AI 理解错误 | AI 识别错业务对象或状态 |
| 断言规则错误 | 断言字段或阈值配置错误 |

判断理由：

普通测试工具只说执行失败。Crab 必须告诉用户失败类型、原因、影响范围和推荐下一步。

### 4.3 AI 解释结构

确认规则：

AI 输出必须包含：

- 依据。
- 结论。
- 置信度。
- 待确认点。
- 风险点。

示例：

```text
依据：PRD 第 3 节提到“管理员可启用被禁用用户”，页面字段包含 enabled。
结论：AI 推测本需求包含 Disabled -> Enabled 状态流转。
置信度：中。
待确认点：enabled 是否是判断用户启用状态的权威字段？
风险点：如果仅验证页面状态，可能无法证明数据层状态已变化。
```

判断理由：

AI 不能只给答案。Crab 里的 AI 输出必须让人知道它根据什么得出结论、它有多确定、哪些地方必须人工确认、哪些地方可能导致证据不可信。

## 5. 设计系统与交付清单

Figma 负责视觉，页面交付说明负责行为、状态、权限、数据和证据链。二者缺一不可。

### 5.1 页面交付标准

每个页面交付必须包含：

| 项 | 内容 |
| --- | --- |
| 页面用途 | 这个页面帮用户完成什么任务 |
| 用户角色 | 谁主要使用 |
| 入口来源 | 从哪里进入 |
| 页面结构 | 顶部、左侧、中间、右侧、底部怎么组织 |
| 字段说明 | 每个字段含义、来源、是否必填 |
| 状态说明 | loading、empty、error、success、disabled |
| 交互说明 | 点击、确认、驳回、重试、重跑、展开、筛选 |
| 异常情况 | 环境失败、权限失败、证据不足等 |
| 权限差异 | owner、member、viewer 等能做什么 |
| 数据需求 | 需要哪些接口 / 数据 |
| 证据关系 | 展示或生成什么证据，如何追溯 |

判断理由：

Crab 页面有很多 Figma 看不出来的规则：按钮什么时候禁用、失败时怎么提示、谁能确认、证据不足时报告怎么显示、哪个数据可以追溯到执行记录。这些必须在页面交付说明里写清楚。

### 5.2 组件交付标准

业务组件交付必须包含：

| 项 | 内容 |
| --- | --- |
| 组件用途 | 解决什么重复业务问题 |
| 输入数据 | 需要哪些字段 |
| 输出动作 | 用户能做什么，会产生什么决策 |
| 状态 | loading、empty、error、disabled、不同业务状态 |
| 文案 | 标题、按钮、tooltip、错误提示 |
| 权限 | 谁能看、谁能点、谁能确认 |
| 证据关系 | 是否关联 Proof Case / Execution / Evidence |
| 基础组件 | 基于 shadcn-vue 哪个 primitive |
| 参考产品 | 参考 Hoppscotch / Playwright / Allure / Dify / Open WebUI / Cline / Supabase 哪个模式 |

判断理由：

Crab 真正的设计资产不是 Button、Input，而是 Proof Case 状态卡、AI 草案卡、人工确认条、证据矩阵、执行时间线、tool call 记录、报告结论卡等业务组件。

## 6. 页面检查表

检查表不是设计完以后随便看一眼，而是每个页面进入开发前必须过一遍。

每个页面至少通过：

1. UX 检查。
2. UI 检查。
3. 视觉检查。
4. 交互检查。
5. 交付检查。
6. 证据可信检查。

### 6.1 UX 检查

- 这个页面服务哪个用户？
- 用户来这里要完成什么任务？
- 用户下一步应该做什么？
- 哪些内容必须人工确认？
- 哪些内容是 AI 建议？
- 失败时用户怎么办？
- 这个页面是否能减少 QA 的确认成本？

### 6.2 UI 检查

- 页面主信息是什么？
- 第一眼应该看到什么？
- 哪些信息默认展示？
- 哪些信息应该折叠？
- 有没有清晰的下一步按钮？
- 表格字段是否过多？
- 状态是否容易识别？

### 6.3 视觉检查

- 字体是否统一？
- 字号层级是否清楚？
- 颜色是否只用于表达状态？
- 间距是否统一？
- 图标风格是否一致？
- 页面是否过度装饰？
- 是否符合专业工具气质？

### 6.4 交互检查

- 点击后有没有反馈？
- 加载中有没有过程说明？
- 失败后有没有原因？
- 能不能撤销？
- 能不能重新生成？
- 能不能重跑？
- 能不能查看原始证据？
- 能不能知道是谁确认的？

### 6.5 交付检查

- 页面用途是否说明清楚？
- 用户角色是否说明清楚？
- 字段来源是否说明清楚？
- 状态、权限、异常、数据需求是否说明清楚？
- 组件输入、输出、状态、文案是否说明清楚？

### 6.6 证据可信检查

证据可信检查优先级最高。如果证据链不清楚，即使页面好看，也不能进入开发。

- 是否区分 AI 建议和人工确认？
- 是否区分测试通过和状态成立？
- 是否展示证据来源？
- 是否展示证据缺口？
- 是否能追溯到执行记录？
- 是否能看到谁确认过？
- 是否能查看原始证据？
- 是否能沉淀为可复用资产？

## 8. 当前确认结论

本轮已经确认：

- 视觉设计清单采用高密度专业工作台方向。
- Codex 风格只用于 AI / Agent 执行层。
- 颜色只服务状态、风险、责任归属。
- 字体字号采用紧凑型专业工具体系。
- 间距、圆角、卡片保持克制。
- 动效只用于功能性状态变化。
- 组件体系分为基础组件、业务组件、证据/状态组件。
- Dialog / Alert Dialog / Sheet / Drawer 有明确边界。
- 高频工作面优先 Data Table。
- Proof / Evidence / AI / Execution 状态统一 Status Badge 体系。
- 文案采用事实型、结论型、可审计风格。
- 错误提示使用四段式结构。
- AI 解释必须包含依据、结论、置信度、待确认点、风险点。
- 页面交付不能只给 Figma 图，必须有页面交付说明。
- 业务组件必须有组件说明。
- 页面进入开发前必须通过 UX、UI、视觉、交互、交付、证据可信六类检查。
- 证据可信检查优先级最高。
